Projects View Redesign — Two Approaches

Both share the same filter bar and data; the difference is layout philosophy

A

Toolbar + Adaptive Grid

Horizontal filter/action bar at the top (Active toggle, system dropdown, tags, search, view toggle). Below it, a responsive CSS grid of compact cards that reflows based on window width. Table view replaces the grid entirely when toggled. Clean, flat, modern — like GitHub repos or Notion databases.

Pros

  • Maximum screen real estate for project cards
  • Familiar pattern (GitHub, Linear, Notion)
  • Scales from 5 to 50+ projects naturally
  • Simple mental model — one area, one set of controls

Cons

  • Filter bar takes vertical space
  • No persistent project detail without clicking into one
  • Less visual hierarchy between active vs inactive
B

Sidebar List + Main Stage

Left sidebar becomes a filterable, scrollable project list (compact rows, not cards). Filter controls live at the top of the sidebar. Right side is the "stage" — selected project's details (tasks, docs, knowledge, extensions). The sidebar stays visible while you work in a project.

Pros

  • Always see your project list while working in one
  • Fast switching — click sidebar item, stage updates
  • Natural for "I have 25 projects and jump between them"
  • Similar to VS Code explorer, Slack channels, email clients

Cons

  • Sidebar takes ~250-300px permanently
  • Project cards lose visual richness (compact list items)
  • Grid/table toggle only applies to the sidebar list
  • More complex layout (sidebar + stage coordination)
